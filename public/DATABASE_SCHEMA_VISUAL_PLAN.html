XMETA Pay Visual Database Schema Plan

A browser-viewable version of the full practical MVP database plan for the admin/school dashboard and parent portal. This file is documentation only and does not change the live SQL import file.

MySQL / XAMPP InnoDB utf8mb4 Admin + Parent Portals

Project Overview

Admin/school portal School setup, student records, parent directory, tuition, collections, allowance, store transactions, and CSV/PDF reports.
Parent portal Registration, login, student linking by reference, linked enrolled student access, fee viewing, tuition payment, receipts, history, wallet top-up, and wallet store-spending history.
Auth foundation Keep the existing shared users table for super_admin, admin, and parent roles, with role-specific admin_profiles and parent_profiles. Admin profiles link to schools.id after manual setup. New school admin users start with users.status = pending until the company super admin approves or rejects them.
Admin staff roles admin_profiles.staff_role controls admin dashboard permissions. See ADMIN_ROLES.md for school_administrator, registrar, and finance_officer.

Core Table Groups

Module Tables Main purpose
Auth users, auth_sessions, admin_profiles, parent_profiles Company super admin login, admin registration approval, portal registration/login, DB-backed sessions, role redirects, and account status.
School setup schools, school_years, grade_levels, sections School structure with one or many school years, one active year, selected-year section editing, manual rollover targets, and activation that closes the previous active year.
Students students, student_guardians Student profiles and parent-to-student links, including one parent linked to multiple students.
Enrollment enrollments, enrollment_documents Year-specific enrollment status, grade/section assignment, and student type (new, transferee, or returned). Student master records also keep reusable sex data; age is derived from birthdate. Rollover and bulk enrollment create only reviewed or selected enrollment rows, never duplicate student identity or guardian-link data.
Fees fee_types, fee_type_term_templates, student_fee_assignments, tuition_payment_terms Tuition, other fees, grade/section-filtered selected-student bulk assignment, due dates, balances, per-student tuition terms, and shared server-side tuition term rules. Fee type term templates remain schema-ready for future reuse.
Payments payments, payment_allocations, payment_term_allocations, receipts Payment records, active school-year stamps for new rows, regular fee allocation, tuition term allocation, and receipt generation.
Wallet wallets, wallet_transactions Student allowance balance, year-stamped top-ups/purchases for new rows, dashboard activity, selected profile activity, and adjustments. Current totals sum each wallet once; admin monthly top-up stats sum current-month top-up ledger rows.
Store store_merchants, store_transactions Implemented Phase 6B tables for canteen and school store spending logs, with active school-year stamps for new purchase rows.
Reports Operational query views Implemented CSV and PDF exports for monthly revenue, tuition collections, outstanding balances, and wallet/store activity. Tuition collections use fee or term allocations; wallet-only payments remain in the allowance ledger. Real-data dashboard tables paginate loaded rows and export filtered rows as CSV or PDF.
Notifications notification_logs Implemented queued SMS/email payment reminder history with saved message_body text, school-year stamps for new rows, and once-per-day deduplication for the same school year, school, linked parent, student, and selected channel. Real delivery, receipts, low-wallet alerts, and enrollment-update notifications remain future.

Parent Wallet and Allowance Flow

Implemented: wallet top-up, parent dashboard wallet activity, selected student profile wallet activity, admin allowance monthly top-up stats, segmented allowance balance filters, and Phase 6B store purchase recording. Admin store actions use focused Create merchant and Record purchase modals above the transaction log. Admin allowance total balance sums current wallet balances once per wallet.

flowchart TD
  A["Parent opens wallet page"] --> B["Load student wallets"]
  B --> C["Parent chooses top-up amount"]
  C --> D["Create payment for wallet top-up"]
  D --> E["Payment succeeds"]
  E --> F["Increase wallet balance"]
  F --> G["Create wallet top_up transaction"]
  G --> H["Admin or finance records canteen/store purchase"]
  H --> I["Create wallet purchase transaction"]
  I --> J["Create store transaction"]
  J --> K["Parent sees wallet history and dashboard wallet activity"]
  K --> K1["Selected student profile shows that student's wallet activity"]
  K --> K2["Admin sees store reports"]
  K --> L["Admin allowance total sums each wallet balance once"]

Wallet transaction rows explain top-ups and purchases. The parent dashboard shows a recent wallet activity snapshot, the selected student profile shows that student's activity, and the wallet page remains the full ledger. Current balance totals should come from wallets.balance, grouped by wallet, so one wallet with multiple transactions is not counted multiple times.

Suggested Implementation Order

  1. Keep the current auth schema working first.
  2. Add school setup tables.
  3. Add student and guardian tables.
  4. Add enrollment tables.
  5. Add fee tables.
  6. Add payment and receipt tables.
  7. Add wallet and wallet transaction tables.
  8. Add store merchant and store transaction tables. These already exist in the schema and support the implemented Phase 6B write flow.
  9. Build CSV/PDF reports from operational query views and CSV/PDF filtered-row table exports from loaded dashboard rows instead of separate report storage tables.
  10. Use notification logs for implemented queued in-app payment reminder history with same-day duplicate prevention; add real email/SMS delivery later.
